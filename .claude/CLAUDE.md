# CLAUDE.md

## 0. 目的

このプロジェクトで Claude Code が作業する際のポリシーを明示する

## 言語ポリシー(必須)

- すべての対話・出力は **日本語** で行うこと
- **Git のコミットメッセージは英語**で記述すること(後述の規約に従う)

## ブラウザ操作

- Web コンテンツの取得・操作は claude-in-chrome を使うこと
- fetch/curl が必要な場合は理由を説明してから実行すること

## Git 運用(必須)

### コミットメッセージ規約

- コミットメッセージは **要約（50 文字程度） + 必要ならボディ**。自動生成時も英文の簡潔さを優先
- コミットメッセージは英語でConventional Commitsに準拠 (例：`feat: ...`, `fix: ...`, `docs: ...`)
  言語の指定があった場合を除き、常に英語で記述すること。
- Pull Requestの説明は日本語で詳細に記述すること
- 変更点が多く、デフォルトブランチへの直 push が不適切な場合は PR 作成を提案すること

### 基本的なpush運用

- 「push して」という指示を受けた場合は、以下の順序で実施すること :
  1. 変更点の確認(`git status` / `git diff`)
  2. ステージング(`git add`)・ すでにステージングされているものがある場合は新たにステージングしない
  3. コミットメッセージ規約に従いコミット
  4. リモートへのプッシュ(デフォルトブランチに直 push が不適切な場合は PR 作成を提案)
- 「commit して」という指示を受けた場合は、以下の順序で実施すること :
  1. 変更点の確認(`git status` / `git diff`)
  2. ステージング(`git add`)・ すでにステージングされているものがある場合は新たにステージングしない
  3. コミットメッセージ規約に従いコミット

### Pull Request作成の自動化

- 「pr作成して」という指示を受けた場合は、以下の手順を自動実行すること :
  1. 現在の変更状況とブランチ構成を確認
  2. 現在のブランチから新しい作業用ブランチを作成(命名規則：`fix/`, `feat/`, `style/`等の接頭辞を使用)
  3. コミットメッセージ規約に従いコミット
  4. 新しいブランチをリモートにプッシュ
  5. 元のブランチに対してPull Requestを作成(日本語の詳細な説明付き)
  6. 元のブランチに戻る
  7. マージ後に作業用ブランチを削除 (またはマージ後の削除を提案)

## セーフティガード

- ファイル編集・依存追加・外部通信は、プロジェクトの既定ルールに従うこと
- 危険と判断した操作は実行前にユーザーに確認を求めること
- ブラウザ自動化の対象サイトは最小限に限定し、個人情報や秘密情報を扱わない

## モデル選択ポリシー

- Haiku: Glob/Grep・検索・lint など推論不要な作業
- Sonnet: デバッグ・実装などに積極的に使用すること
- Opus: 設計・大規模リファクタリング・プロジェクト全体の分析
- 失敗したタスクは上位モデルで再試行すること
- コストと実装速度を考慮して、適切なモデルを選択すること

## SubAgents / AgentTeamsの使用

### デフォルト: SubAgent

- 実装速度を重視し、独立したタスクには SubAgent を積極的に活用すること
- 使用するモデルは、タスクの性質と複雑さに応じてモデル選択ポリシーに従うこと
- 並列実行可能なタスクは並列で SubAgent を起動すること

### AgentTeams(tmux)を使うケース

以下のいずれかに該当する場合は AgentTeams("teammateMode": "tmux") を使用すること:

- タスク間で**相互依存・調整**が必要な場合（仕様検討→実装→テスト→レビューの役割分担）
- **レビュー・フィードバックループ**が有効な場合（実装者とレビュアーの往復）
- **横断的整合性**の維持が必要な場合（複数ファイル/モジュールの一貫性）
- 共有タスクを使った**シーケンシャル/パラレルのパイプライン**実行が有効な場合
- チームメンバーへの**直接介入**(SendMessage)が必要な場合
